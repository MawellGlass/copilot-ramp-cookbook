# Copilot Ramp Cookbook

> A public, community-built guide to getting started with Microsoft Copilot — and growing from your
> first chat all the way to building production agents in Copilot Studio.
>
> **⚠️ Unofficial.** This is a personal/community project. It is **not affiliated with, endorsed by,
> or an official resource of Microsoft.** Product names and trademarks belong to Microsoft.

---

## The journey

```
Copilot Chat → First-Party Agents → Cowork → Agent Builder → Copilot Studio → Microsoft Foundry
```

Most people stall at "I have Copilot — now what?" This cookbook lays out a natural, grassroots ramp:
start by *using* Copilot Chat, get comfortable letting prebuilt agents and Cowork do work for you,
then take the leap from *using* agents to *building* them — first with Agent Builder, then Copilot
Studio (the low-code maker destination), and finally Microsoft Foundry (the pro-code frontier).

Each stage has: an **overview**, **use cases** (filterable by your role), and **curated resources**.

## What's in this repo

The published website is built from the `docs/` folder with **MkDocs Material**; everything else is
planning, tooling, and config.

| File / folder | What it is |
|---------------|-----------|
| `docs/` | **The site source** — all Markdown MkDocs publishes (home, start-here, stage overviews, walkthroughs, catalog, resources, about) |
| `docs/index.md` · `docs/start-here.md` · `docs/about.md` | Home, the "find your rung" self-assessment, and the disclaimer page |
| `docs/stages/` | The seven stage-overview pages (`stage-1-chat.md` … `stage-7-foundry.md`) |
| `docs/walkthroughs/` | The 70+ rich, fully-written walkthroughs (start with `chat-meeting-followups.md`, the exemplar) |
| `docs/solutions/` | Nine ready-to-adapt solution templates across Copilot Studio and Microsoft Foundry |
| `docs/CATALOG.md` | The full use-case matrix — every walkthrough mapped across the seven stages and six roles |
| `docs/RESOURCES.md` | Curated, verified links to official Microsoft resources, organized by stage |
| `docs/screenshots/` | Captured PNGs, one subfolder per walkthrough (generated by the tooling, not hand-made) |
| `mkdocs.yml` | Site configuration — Material theme, journey-first nav, search, tag filtering, light/dark toggle |
| `requirements.txt` | Pinned build dependency (`mkdocs-material`) so local and CI builds match |
| `.github/workflows/deploy.yml` | GitHub Action that builds and publishes the site to GitHub Pages on every push to `main` |
| `GO-PUBLIC.md` | Step-by-step runbook: local preview → git init → push → enable Pages → deploy |
| `PLAN.md` | The build plan: information architecture, the journey × role matrix, hosting recommendation, phases |
| `CONTENT-MODEL.md` | The locked spec — frontmatter schema, the rich-walkthrough template, and authoring rules |
| `tooling/screenshots/` | Playwright tool that drives the real product and captures numbered screenshots for the docs |
| `mockup/` | A standalone HTML design mockup of the site look-and-feel (not part of the build) |

## Who it's for

End users who want a first win · Champions rolling Copilot out to a team · Managers building the case ·
Makers ready to build their first agent · IT/admins enabling each stage safely.

## Status

**Live and maintained.** The seven-stage ramp, 70+ walkthroughs, nine solution templates, and the
connective site pages (home, start-here, stage overviews, about) are published with MkDocs Material
on GitHub Pages. Screenshots are two-state: a walkthrough either embeds real captures generated by the
Playwright tool (committed under `docs/screenshots/`) or carries an honest no-screenshot note pointing
back to the numbered steps we keep current — we never ship faked or stale UI images.

## Build & preview it

```bash
pip install -r requirements.txt
mkdocs serve        # → http://127.0.0.1:8000
```

Full publish steps (git init → push → enable Pages → automatic deploy) are in **`GO-PUBLIC.md`**.

## How to contribute

Content is plain Markdown with frontmatter (see `CONTENT-MODEL.md`). Add or expand a use case by
following the template under `docs/walkthroughs/`, then open a pull request. Keep every prompt
copy-pasteable and every page honest about limits.
